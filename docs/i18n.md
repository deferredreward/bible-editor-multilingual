# UI localization

The Bible Editor chrome (buttons, tooltips, dialogs, toasts, banners) is
localized with [i18next](https://www.i18next.com/) + `react-i18next`. Scripture
content itself is project data, not UI — this doc is only about the interface.

## Where things live

| Path | What |
| --- | --- |
| `web/src/i18n/index.ts` | i18next init + the `UI_LANGUAGES` registry (code, native label, `dir`) |
| `web/src/i18n/locales/en.json` | **Source of truth** — every key the UI can request |
| `web/src/i18n/locales/<code>.json` | One file per language, mirroring `en.json` |
| `scripts/check-i18n.mjs` | Completeness checker (run in CI / before a PR) |

Keys are namespaced (`common`, `topbar`, `translation`, `export`,
`importDialog`, `questions`, `words`, `rail`, `sync`, `shell`, `pipeline`) and
referenced in code as `t("ns.key")`. `en` is the `fallbackLng`, so a missing
key in another locale silently renders English rather than a raw `ns.key` —
which is why the checker exists: fallbacks hide gaps.

## Adding a language

1. **Register it** in `web/src/i18n/index.ts`:
   - `import xx from "./locales/xx.json";`
   - add `xx: { translation: xx }` to the `resources` block;
   - add a row to `UI_LANGUAGES` with the **native** label and the correct
     `dir` (`"rtl"` for right-to-left scripts — Arabic, Farsi, Urdu, Hebrew;
     `"ltr"` otherwise). `dir` drives full mirroring: `document.dir`, the MUI
     theme direction, and the emotion RTL cache all key off it (see `main.tsx`).
2. **Copy the full key set** from `en.json` into `web/src/i18n/locales/xx.json`
   — same namespaces, same keys, same nesting.
3. **Translate the values.** Two hard rules:
   - **Preserve `{{var}}` placeholders exactly** (`{{book}}`, `{{count}}`,
     `{{message}}`, …). They are interpolated at runtime; renaming or dropping
     one breaks the message.
   - **Keep Latin resource tokens literal:** `ULT`, `UST`, `UHB`, `UGNT`,
     `Door43`, `DCS`, `TW`, `TN`, `TQ`, `TWL`, and key hints like `Ctrl+F`.
     These are proper names / shortcuts, not words to translate.
4. **Get the plurals right.** i18next resolves a base key through suffixed
   variants using the CLDR plural rules for that language (via
   `Intl.PluralRules`). Provide exactly the categories your language uses:

   | Language(s) | Categories | Suffixes to supply |
   | --- | --- | --- |
   | id, th, zh, ja, ko | `other` | `_other` |
   | es, fr, pt | `one, many, other` | `_one _many _other` |
   | hi, sw, ne, bn, ur, fa, tr | `one, other` | `_one _other` |
   | ru, uk (Slavic) | `one, few, many, other` | `_one _few _many _other` |
   | ar (Arabic) | `zero, one, two, few, many, other` | all six |

   The 10 plural base keys today are `sync.conflict`, `sync.queuedLocally`,
   `sync.savingTooltip`, `sync.unsavedEditsColon`, `sync.unsavedEditsClickToJump`,
   `sync.resolveConflicts`, `sync.discardConfirmTitle`, `shell.colCount`,
   `shell.editLeftUnaligned`, `pipeline.pipelinesRunning`.

   > **Note on Farsi (fa) and Urdu (ur):** despite being written in the
   > Arabic script, CLDR classifies both as **one/other** — not the six-way
   > Arabic split. `Intl.PluralRules` (what i18next uses) only ever requests
   > `_one`/`_other` for them, so extra `_zero/_two/_few/_many` keys would be
   > dead weight. Only `ar` itself uses all six.

5. **Verify:** `node scripts/check-i18n.mjs`. It must report your locale as
   complete (exit 0).

## What the checker enforces

`node scripts/check-i18n.mjs` compares every `locales/*.json` (except `en.json`)
against `en.json` and, per locale, reports:

- **MISSING** — base keys in `en.json` absent from the locale. Plural-aware: any
  suffixed variant (`_one`, `_other`, `_zero`, `_two`, `_few`, `_many`)
  satisfies its base key.
- **PLURAL categories missing** — a plural base is present but lacks a category
  that `Intl.PluralRules(<code>)` will actually request for that language (e.g.
  a Russian file with `_one/_other` but no `_few`/`_many`).
- **STALE** — keys in the locale that no longer exist in `en.json` (typos,
  renamed keys, leftover stub `_comment`s).

It exits non-zero if **any** locale has a gap, so it can gate CI and PRs.

## Translation quality

The locale files beyond English and Arabic were **machine-drafted** and have
**not** yet had native-speaker review. They are structurally complete and safe
to ship (placeholders and plurals are validated), but wording, tone, and
terminology need a native pass before they should be considered
production-final. Track native review per language before promoting.
