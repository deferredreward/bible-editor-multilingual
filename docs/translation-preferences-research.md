# Translation Preferences & Memory Management — Industry Research (Phase 1)

**Purpose:** Ground the design of a "translation preferences and memory management" control panel for the bible-editor tool in real, sourced translation-industry practice — not in imitation of any single competitor tool. This document surveys translation memory (TM) conventions, terminology management standards, localization style-guide/brief practice, CAT-tool QA taxonomies, LQA error typologies (MQM/DQF), how modern MT/LLM pipelines are steered with glossaries and examples, and Bible-translation-specific incumbent practice (Paratext, unfoldingWord).

**Method:** Web search + primary-source fetch (standards bodies — ISO, TBX/NISO-STS; vendor documentation — RWS/Trados, memoQ, Smartling, Crowdin, DeepL, AWS, Google Cloud; academic/industry bodies — TAUS, MQM Council; and unfoldingWord's own Gateway Language Manual, translationWords repo, and Paratext manual). Every load-bearing claim below is cited inline with a markdown link; all sources were accessed **2026-07-13**. Where a claim could not be substantiated, it is flagged in the **Gaps** section rather than invented.

## 1. Translation memory (TM) conventions

### 1.1 TMX as the exchange format

**Claim:** TMX (Translation Memory eXchange) is the de facto standard for exchanging translation memories between CAT tools; the current stable version is 1.4b, and it is an XML vocabulary maintained originally under LISA/OSCAR and now hosted by GALA.
**Evidence:** The TMX 1.4b specification itself, and secondary confirmation via Wikipedia. ([TMX 1.4b Specification](https://www.ttt.org/oscarStandards/tmx/tmx14b.html), accessed 2026-07-13; [Translation Memory eXchange — Wikipedia](https://en.wikipedia.org/wiki/Translation_Memory_eXchange), accessed 2026-07-13)

**Structure — `<header>`:**
Required attributes: `creationtool`, `creationtoolversion`, `segtype`, `o-tmf`, `adminlang`, `srclang`, `datatype`. Optional: `o-encoding`, `creationdate`, `creationid`, `changedate`, `changeid`. The header carries document-level metadata and can hold `<note>`, `<prop>`, and `<ude>` (user-defined encoding) child elements. ([TMX 1.4b Specification](https://www.ttt.org/oscarStandards/tmx/tmx14b.html), accessed 2026-07-13)

**Structure — `<tu>` (translation unit):**
A `<tu>` is the container for one source segment and all its translated variants. Optional attributes: `tuid` (unique identifier), `srclang` (overrides header default), `datatype`, `usagecount` (how often the unit has been reused/leveraged), `lastusagedate` (ISO 8601 timestamp), `creationtool`/`creationtoolversion`, `creationdate`/`creationid`, `changedate`/`changeid`, `segtype` (`block` | `paragraph` | `sentence` | `phrase`), `o-tmf` (originating TM format). ([TMX 1.4b Specification](https://www.ttt.org/oscarStandards/tmx/tmx14b.html), accessed 2026-07-13)

**Structure — `<tuv>` (translation unit variant):**
Each `<tu>` holds at least one `<tuv>` — the segment text in one specific language. Required attribute: `xml:lang` (per RFC 3066/BCP-47). Optional attributes mirror most of the `<tu>` set (`o-encoding`, `datatype`, `usagecount`, `lastusagedate`, `creationtool`/version, `creationdate`/`creationid`, `changedate`/`changeid`, `o-tmf`). The `<tuv>` holding the source-language segment has its `xml:lang` set to the same value as the header/tu's `srclang`. ([TMX 1.4b Specification](https://www.ttt.org/oscarStandards/tmx/tmx14b.html), accessed 2026-07-13; corroborated by [Translation Memory eXchange — Wikipedia](https://en.wikipedia.org/wiki/Translation_Memory_eXchange), accessed 2026-07-13)

**Note on the spec's own scope:** TMX 1.4b itself defines no match-quality scoring or penalty mechanism — it is a pure exchange format. It relies on `<bpt>`/`<ept>` (begin/end paired tags) for inline formatting preservation, and on `usagecount`/`lastusagedate` as raw signals a *consuming* tool can use to infer relevance. Fuzzy-match scoring and penalties are implemented by individual CAT tools on top of TMX-imported data, not by the format itself. ([TMX 1.4b Specification](https://www.ttt.org/oscarStandards/tmx/tmx14b.html), accessed 2026-07-13)

### 1.2 Match types and thresholds — the vendor-level convention

Although TMX doesn't define these, there is strong cross-vendor convergence on a match taxonomy:

- **100% match:** the source segment is character-for-character identical (including formatting/tags) to a stored TM segment. ([Understanding 100% matches, context matches, fuzzy matches...](https://knowledgebase.pairaphrase.com/knowledge-base/what-are-100-matches-context-matches-and-fuzzy-matches), accessed 2026-07-13)
- **101%/102% or "Context Match"/"ICE" (In-Context Exact) match:** text is identical *and* the surrounding context (preceding/following segment, document structure) also matches, so the tool treats it as needing no review at all. memoQ calls this an **ICE match**; Trados Studio calls the equivalent **Context Match**, and additionally has **PerfectMatch**, which compares a *new* source file against a set of already-translated bilingual files (not the TM) and requires surrounding-segment context to match — often surfaced as 101%. Vendors disagree on numbering: memoQ's 102% ≈ Trados's PerfectMatch; memoQ's 101% ≈ Trados's Context Match. ([Context Match Explained — Rankvise](https://rankvise.com/blog/context-match/), accessed 2026-07-13; [About Perfect Match — RWS/Trados docs](https://docs.rws.com/en-US/trados-studio-2022-sr2-1107960/about-perfect-match-341156), accessed 2026-07-13; [Context Match vs. Perfect Match — Coruzant](https://coruzant.com/tech/context-match/), accessed 2026-07-13). The underlying "In-Context Exact (ICE) Matching" concept is also the subject of US patents held by SDL/RWS. ([US20100262621A1 — In-context exact (ice) matching, Google Patents](https://patents.google.com/patent/US20100262621A1/en), accessed 2026-07-13)
- **Fuzzy matches (below 100%):** partial similarity. Configurable minimum threshold (commonly defaulted around 70–75%); matches below ~50% are usually treated as no-match/new-segment because the reusable value is too low. High fuzzy (85–95%) typically differs by about one word for an average-length segment. Tools let admins configure the threshold plus **penalties** — point deductions applied for differences in numbers, dates, tags, formatting, or for matches sourced from a lower-trust TM. ([The Fuzziness of Fuzzy Matches — TM-Town](https://www.tm-town.com/blog/the-fuzziness-of-fuzzy-matches), accessed 2026-07-13; [Types of translation memory matches — RWS WorldServer docs](https://docs.rws.com/en-US/worldserver-11-8-user-documentation-975503/types-of-translation-memory-matches-305144), accessed 2026-07-13; [How do you optimize TM and terminology for fuzzy match — LinkedIn Advice](https://www.linkedin.com/advice/0/how-do-you-optimize-your-translation-memory-terminology), accessed 2026-07-13)

### 1.3 TM maintenance — penalizing/retiring bad units

Common, cross-source-documented practices:

- **Pruning stale entries:** periodically remove segments unused for years, or tied to retired products/campaigns. ([Translation memory maintenance — POEditor Blog](https://poeditor.com/blog/translation-memory-maintenance/), accessed 2026-07-13; [How to maintain a clean translation memory — Gridly](https://www.gridly.com/blog/maintain-clean-translation-memory-sanity-guide-localization-teams/), accessed 2026-07-13)
- **Deduplication/health checks:** quarterly-cadence removal of redundant identical-source segments and numbers/punctuation-only noise segments. ([How to maintain a clean translation memory — Gridly](https://www.gridly.com/blog/maintain-clean-translation-memory-sanity-guide-localization-teams/), accessed 2026-07-13)
- **Penalizing lower-quality/lower-trust TMs:** when multiple TMs feed one project (e.g. a legacy TM vs. a current-style TM), apply a **penalty** (point deduction) to matches sourced from the lower-trust TM so they surface as fuzzy rather than perfect matches and get human review. ([Best Practices in Translation Memory Management — Alphatrad](https://www.alphatrad.net/best-practices-in-translation-memory-management/), accessed 2026-07-13; [TM maintenance and TM clean-up — Loctimize](https://www.loctimize.com/en/support/tm-maintenance-tm-clean-up), accessed 2026-07-13)
- **Annual deep optimization:** align legacy TM content with current terminology/brand voice; this is explicitly described as a "stated plan or guiding principles for grouping, priority, applying, penalization, and updating of TMs." ([Translation Memory Management: Optimization & Best Practices — Translated](https://translated.com/resources/translation-memory-management-optimization-best-practices), accessed 2026-07-13)

## 2. Terminology management

### 2.1 TBX as the standard, and its dialects

**Claim:** TBX (TermBase eXchange) is the ISO standard (ISO 30042) for exchanging **concept-oriented** terminological data — i.e., entries are organized around a *concept*, with one or more *terms* per language attached to that concept, each carrying its own metadata, rather than being organized as flat word-pair lists.
**Evidence:** ISO 30042:2019 supersedes ISO 30042:2008; TBX was co-published by ISO and LISA. ([ISO 30042:2019 — Management of terminology resources — TermBase eXchange (TBX)](https://www.iso.org/standard/62510.html), accessed 2026-07-13; [ISO 30042:2008](https://www.iso.org/standard/45797.html), accessed 2026-07-13; [TermBase eXchange — Wikipedia](https://en.wikipedia.org/wiki/TermBase_eXchange), accessed 2026-07-13)

**Dialects:** Historically, **TBX-Basic** (published 2008 by the LISA Terminology SIG) and **TBX-Min** (published 2013 by LTAC Global) were separate lightweight profiles. **TBX Version 3** (folded into ISO 30042:2019) recast Min and Basic as *modules* built on a shared **TBX-Core**, with an extensibility mechanism (XCS — eXtensible Constraint Specification) for validating dialects and constraining which data categories/picklist values a given termbase file may use. ([TBX Version 3 published at ISO — MultiLingual](https://multilingual.com/issues/july-aug-2019/tbx-version-3-published-at-iso/), accessed 2026-07-13; [TBX Namespaces — tbxinfo.net](https://tbxinfo-c.byuling.net/tbx-namespaces/), accessed 2026-07-13)

### 2.2 Term-level status: the actual ISO/TBX picklist

**Claim:** TBX's `normativeAuthorization` (administrative status) data category is a closed picklist, not free text.
**Evidence — the actual values**, from the ISO/NISO-STS TBX schema documentation: `preferredTerm-admn-sts`, `admittedTerm-admn-sts`, `deprecatedTerm-admn-sts`, `supersededTerm-admn-sts`, `legalTerm-admn-sts`, `regulatedTerm-admn-sts`, `standardizedTerm-admn-sts`. ([tbx:normativeAuthorization-attlistT — ISO schema docs](https://www.iso.org/schema/nisosts/v0.2/doc/tbx/ISO-TBX_xsd_Attribute_Group_tbx_tbx_normativeAuthorization-attlist.html), accessed 2026-07-13; [Element: Normative Authorization Flag — NISO-STS](https://niso-sts.org/tbx-html/element/tbx-normativeAuthorization.html), accessed 2026-07-13)

Definitions (from the same schema family and corroborating standards documentation):
- **preferred** — "intrinsically better" or the term chosen for consistency; should predominate in body text. ([tbx:normativeAuthorization — ISO schema docs](https://www.iso.org/schema/nisosts/v0.2/doc/tbx/ISO-TBX_xsd_Element_tbx_normativeAuthorization.html), accessed 2026-07-13)
- **admitted** — correct but a better synonym exists; usable, but less often than preferred. (Same source.)
- **deprecated** — widely used and likely to appear in documents, but should not be used and should be changed on editing — i.e., "not recommended." (Same source.)
- **superseded** — was correct/current at one point but has been replaced. (Same source, listed alongside deprecated/obsolete framing.)
- An entry with only one term for a language is *automatically* preferred; no explicit tag is required. ([Element: Normative Authorization Flag — NISO-STS](https://niso-sts.org/tbx-html/element/tbx-normativeAuthorization.html), accessed 2026-07-13)

**Important nuance for design:** TBX's own standard picklist does **not** include a "forbidden" value — "forbidden" is a convention that individual CAT tools (memoQ, Trados) layer on top as a proprietary flag, distinct from the TBX-standard `deprecated`/`superseded` categories. See 2.3.

### 2.3 "Forbidden term" pattern in CAT tools, and DNT lists

- **memoQ:** term bases can mark entries "forbidden," and forbidden-term hits are visually distinguished (e.g., shown in black) from approved glossary matches (blue), separate from ordinary preferred/synonym terms; forbidden terms trigger immediate QA flags. ([Terminology management in memoQ 8.3 — memoQ blog](https://blog.memoq.com/terminology-management-in-memoq-8-3), accessed 2026-07-13; [memoQ, SDL Trados, OmegaT — Translation Geek](https://translationgeek.de/blog/memoq-sdl-trados-omegat-what-is-a-cat-tool-and-do-you-need-one/), accessed 2026-07-13)
- **General CAT-tool convention:** visual/UI distinction between **mandatory** (must-use), **preferred** (recommended), and **forbidden** (never-use) terms; forbidden-term detection is one of the standard QA checks (see §4). ([Translation Glossaries & Termbases: The 2026 Guide — Crisol](https://www.crisoltranslations.com/our-blog/translation-glossaries-termbases/), accessed 2026-07-13)
- **DNT (Do-Not-Translate) lists:** a distinct convention from "forbidden" — a DNT flag tells the translator to leave the source term as-is (untranslated) rather than substituting anything; common DNT entries are registered trademarks, product names, technical standard names, and format names. This is functionally a term-status value of its own (`do-not-translate`), separate from preferred/admitted/deprecated/forbidden. (Same source — Crisol glossary guide, accessed 2026-07-13)
- **The forbidden→preferred pairing pattern:** the recurring UX pattern across tools is: forbidden-term entries carry a **paired "use instead" / preferred-replacement field**, so a QA flag on a forbidden term can immediately surface what to use instead rather than just saying "don't use this." (Synthesized from memoQ/Trados terminology-management documentation cited above; this specific pairing-field convention is common practice but no single vendor doc enumerates it as a named schema field — flagged with **moderate confidence**, not a hard standard citation.)

### 2.4 CSV glossary import conventions

Cross-vendor convergence (DeepL, Google Cloud Translation Advanced, Amazon Translate, Smartcat, and generic CAT tooling):
- Plain **CSV or TSV**, UTF-8 encoded, with a **header row** naming each column (language code or language name, plus attribute columns like Definition/Gender/Context/Comment). ([CSV Format to Load Glossaries and Terms — Embarcadero/Connect docs](https://docwiki.embarcadero.com/Connect/en/CSV_Format_to_Load_Glossaries_and_Terms), accessed 2026-07-13)
- Google Cloud Translation Advanced accepts **TSV, CSV, or TMX** for glossaries; each TSV/CSV row is one term pair separated by tab or comma; it supports both **unidirectional** glossaries (one source→target pair) and **equivalent term sets** (the same concept's term in N languages). ([Creating and using glossaries (Advanced) — Google Cloud docs](https://docs.cloud.google.com/translate/docs/advanced/glossary), accessed 2026-07-13)
- Amazon Translate custom terminology similarly accepts **CSV, TSV, or TMX**, with multidirectional terminology support added in Nov 2021 (one file usable across multiple source/target language pairs). ([Creating a custom terminology — Amazon Translate docs](https://docs.aws.amazon.com/translate/latest/dg/creating-custom-terminology.html), accessed 2026-07-13; [Amazon Translate now enables multidirectional custom terminology — AWS](https://aws.amazon.com/about-aws/whats-new/2021/11/amazon-translate-enables-multidirectional-custom-terminology), accessed 2026-07-13)
- Convention: keep source term **immediately adjacent to** its target term column-wise; attribute columns follow the term column they describe; don't mix language-code conventions (e.g. "EN" vs "EN-US") within one file. ([Managing terminology in Smartcat — LinkedIn](https://www.linkedin.com/pulse/why-smartcat-how-import-xlsx-glossary-joshua-d-vel%C3%A1squez), accessed 2026-07-13)

## 3. Style guides & translation briefs in localization

### 3.1 ISO 17100 and the translation brief

**Claim:** ISO 17100 requires translation service providers to formally capture **project specifications** before production — this is the standards-body backing for what the industry colloquially calls a "translation brief."
**Evidence:** ISO 17100 requires defining specifications that "can include language combinations, deadline, delivery format, purpose of the text, target audience, terminology, style guide, reference material and confidentiality requirements." It structures work into Resources / Pre-Production / Production / Post-Production phases, and explicitly lists value-added services (localization, terminology management, transcription, subtitling, revision, back-translation, DTP, cultural adaptation) that may accompany translation. ([Understanding ISO 17100 — EC Innovations](https://www.ecinnovations.com/blog/understanding-iso-17100-key-principles-for-translation-quality/), accessed 2026-07-13; [ISO 17100:2015 — ISO.org](https://www.iso.org/standard/59149.html), accessed 2026-07-13; [ISO 17100 — Wikipedia](https://en.wikipedia.org/wiki/ISO_17100), accessed 2026-07-13)

**Common LSP practice for brief contents** (industry-consensus, non-normative but widely converged upon across LSP blogs): project name; source/target language(s); **target audience** (demographics, expertise level, context of use); **purpose** (inform / persuade / instruct / comply); delivery channel; **tone and register** (formal / semi-formal / conversational / technical); terminology requirements; certification needs; reference materials; formatting requirements; deadline; review process. Projects with a structured brief are reported to need 30–50% fewer revision rounds. ([How to Write a Translation Brief That Gets Better Results — Lexigo](https://lexigo.com/post/how-to-write-translation-brief-better-results), accessed 2026-07-13; [Creating a Killer Translation Brief — GetBlend](https://www.getblend.com/blog/creating-translation-brief/), accessed 2026-07-13; [The Importance of a Good Translation Brief — Argo Translation](https://www.argotranslation.com/blog/the-importance-of-a-good-translation-brief), accessed 2026-07-13)

### 3.2 Public localization style guides — structure

- **Microsoft Writing Style Guide** (public at learn.microsoft.com): defines voice ("warm and relaxed, crisp and clear, ready to lend a hand"), plus a separate page enumerating **Microsoft Localization Style Guides** per target-language market (general localization guidance, language-specific style/usage in technical documentation, and market-specific data formats), backed by the **Microsoft Terminology Collection** (a searchable, downloadable approved-terminology database per product/language). For localization specifically, the guide's explicit advice: use **one word consistently per concept**, avoid synonyms for the same concept, don't reuse one word for multiple concepts, and avoid idioms/colloquialisms/culture-specific references because they're hard to both understand cross-culturally and to machine/human translate. ([Welcome — Microsoft Writing Style Guide](https://learn.microsoft.com/en-us/style-guide/welcome/), accessed 2026-07-13; [Microsoft Localization Style Guides — Globalization docs](https://learn.microsoft.com/en-us/globalization/reference/microsoft-style-guides), accessed 2026-07-13; [Top 10 tips for Microsoft style and voice](https://learn.microsoft.com/en-us/style-guide/top-10-tips-style-voice), accessed 2026-07-13)
- **Google developer documentation style guide** (public at developers.google.com/style): structured as About/Highlights/**Word list** (a controlled list of spelling/hyphenation/usage decisions for recurring terms, falling back to Merriam-Webster if a term isn't listed) plus a dedicated **Voice and tone** page (conversational, second person, present tense, active voice, serial comma; "sound like a knowledgeable friend," not overly formal or overly entertaining). ([About this guide](https://developers.google.com/style), accessed 2026-07-13; [Voice and tone](https://developers.google.com/style/tone), accessed 2026-07-13; [Word list](https://developers.google.com/style/word-list), accessed 2026-07-13)

These two public guides converge on the same structural template a style-guide feature in a translation tool would need to encode: (a) a **voice/tone statement**, (b) a **controlled word list** (spelling/usage decisions, i.e. a lightweight terminology layer distinct from a full termbase), and (c) **localization-specific rules** (idiom avoidance, one-term-one-concept, market-specific formatting).

### 3.3 Formality as a first-class MT parameter

Modern MT/LLM APIs already expose **formality** as a structured, closed-value control — directly relevant prior art for a register/formality preference field:
- **DeepL:** `formality` parameter with values `default` (auto-detect from source), `more` (formal), `less` (informal), plus graceful-fallback variants `prefer_more`/`prefer_less` that don't error if the target language lacks formality support. Requires a Pro plan; only available for some target languages. ([About the formality feature — DeepL Help Center](https://support.deepl.com/hc/en-us/articles/4406432463762-About-the-formality-feature), accessed 2026-07-13; [Availability of the formal/informal feature — DeepL](https://support.deepl.com/hc/en-us/articles/4406432804498-Availability-of-the-formal-informal-feature), accessed 2026-07-13)
- **Amazon Translate:** formality customization for six target languages (French, German, Hindi, Italian, Japanese, Spanish) with three options — **Default** (no control), **Formal** (example use case: insurance/healthcare), **Informal** (example use case: gaming/social media). ([Customizing your translations with Amazon Translate](https://docs.aws.amazon.com/translate/latest/dg/customizing-translations.html), accessed 2026-07-13; [Control formality in machine-translated text using Amazon Translate — AWS ML blog](https://aws.amazon.com/blogs/machine-learning/control-formality-in-machine-translated-text-using-amazon-translate), accessed 2026-07-13)
- **Google Cloud Translation:** confirmed to have an Advanced-edition **glossary** mechanism (TSV/CSV/TMX import, uni- or bi-directional) and a newer **Translation LLM (TLLM)** model; a search specifically for a Google formality *parameter* analogous to DeepL's/Amazon's did **not** surface one in current documentation (see Gaps, §8). ([Creating and using glossaries (Advanced) — Google Cloud](https://docs.cloud.google.com/translate/docs/advanced/glossary), accessed 2026-07-13; [Translation LLM (TLLM) — Google Cloud docs](https://docs.cloud.google.com/translate/docs/translation-llm), accessed 2026-07-13)

### 4.2 Severity levels and waiver/ignore workflows

- **memoQ:** lets admins define categories/subcategories of errors (terminology, punctuation, numbers, etc.) and assign a **severity** (e.g. critical/major/minor) with associated **penalty points**; separately configurable as either a "warning" or an "error." ([memoQ vs Trados Studio comparison — ChatsControl](https://chatscontrol.com/blog/memoq-vs-trados-studio-freelancer-comparison-2026), accessed 2026-07-13)
- **Smartling:** explicit **two-tier severity model** — **Medium severity**: shown as a warning; translator must fix-or-accept before saving/submitting (i.e., can be overridden/accepted). **High severity**: blocks save/submit entirely; cannot be ignored. Low/medium severity issues can be dismissed via an "ignore" (X) control in the Quality Checks panel. ([Quality Checks On Your Work — Smartling](https://help.smartling.com/hc/en-us/articles/10447325366043-Quality-Checks-On-Your-Work), accessed 2026-07-13; [Quality Check Profiles — Smartling](https://help.smartling.com/hc/en-us/articles/360041858653-Quality-Check-Profiles), accessed 2026-07-13)
- **Crowdin:** per-check-type binary configuration of **Error** vs. **Warning**. Warning = translator is notified with fix suggestions but can still "Save anyway." Error = save is blocked until resolved. Also supports a persistent spellcheck **ignore list**. ([QA Check Settings — Crowdin Docs](https://support.crowdin.com/project-settings/qa-checks/), accessed 2026-07-13; [Guide to Translation QA with QA Checks — Crowdin blog](https://crowdin.com/blog/qa-checks-article), accessed 2026-07-13)
- **Convergent pattern:** the industry norm is effectively **2–3 severities** (info/warning shown-but-overridable vs. error/blocking), with an explicit **override/ignore action** for non-blocking severities, and in memoQ's case a **penalty-point** system underlying the severity so aggregate scores can be computed for QA reporting.

## 5. LQA error typologies (MQM / DQF)

### 5.1 MQM — Multidimensional Quality Metrics

**Claim:** MQM is a hierarchical error typology, organizing errors under top-level "dimensions" with specific sub-error types beneath each, plus an orthogonal severity scale.

Top-level dimensions cited consistently across sources: **Accuracy** (mistranslation, overtranslation, undertranslation, addition, omission, untranslated content), **Fluency**/Linguistic Conventions (grammar, punctuation, spelling, unintelligibility), **Terminology** (incorrect/inconsistent term usage, deviation from domain glossary), **Style** (tone/register appropriateness), **Locale convention**, and **Non-translation** (a catch-all for output so garbled it can't be scored on finer dimensions). ([Multidimensional Quality Metrics (MQM) Framework — Emergent Mind](https://www.emergentmind.com/topics/multidimensional-quality-metrics-mqm), accessed 2026-07-13). Note: emergentmind cites this typology to Freitag et al. 2021 (an academic MT-evaluation paper applying MQM), not directly to the MQM Council's canonical page — the canonical typology page ([themqm.org/error-types-2/typology/](https://themqm.org/error-types-2/typology/)) returned HTTP 403 to automated fetch during this research and could only be corroborated via secondary sources; treat the exact sub-error enumeration as **medium confidence**.

**Severity model:** MQM's default scoring uses **severity levels with multipliers** — commonly cited as **Minor = 1, Major = 5, Critical/Non-translation = 25** (a 0-1-5-25 scale including a zero-weight "Neutral" tier for non-penalized notes). ([Multidimensional Quality Metrics (MQM) Framework — Emergent Mind](https://www.emergentmind.com/topics/multidimensional-quality-metrics-mqm), accessed 2026-07-13). Scores aggregate as a weighted sum of error counts × severity multiplier, normalized against segment/word count, with some newer proposals using a non-linear exponential quality function. (Same source.)

### 5.2 DQF-MQM harmonization

**Claim:** TAUS's Dynamic Quality Framework (DQF) and the EU/QT21-project MQM were independently developed and then formally harmonized in June 2015 (TAUS + DFKI, funded via the EU QT21 project), producing a shared error typology usable across both frameworks' tooling. Since 2018, the DQF-compatible subset became known as **"MQM Core."**
**Evidence:** ([DQF and MQM Harmonized to Create an Industry-Wide Quality Standard — TAUS](https://o.taus.net/academy/news/press-release/dqf-and-mqm-harmonized-to-create-an-industry-wide-quality-standard), accessed 2026-07-13; [DQF-MQM error typology — Translators without Borders community forum](https://community.translatorswb.org/t/dqf-mqm-error-typology/10827), accessed 2026-07-13)

The harmonized model, per the TWB community forum's applied description, uses **five dimensions** (Accuracy, Fluency, Terminology, Style, Design) and **four+ severity tiers**: **Critical** ("significant impact, may cause severe implications"), **Major** ("considerable impact, may confuse or mislead the reader"), **Minor** ("slight impact, does not cause loss of meaning"), plus non-penalizing **Neutral** and **Kudos** tiers for stylistic notes/exceptional work. ([DQF-MQM error typology — TWB community forum](https://community.translatorswb.org/t/dqf-mqm-error-typology/10827), accessed 2026-07-13) — this specific forum post did not state numeric multipliers, so treat DQF's exact point-weighting as **not verified** (see Gaps).

### 5.3 How human-correction classification feeds process improvement

The general mechanism documented across MQM/DQF sources: individual **post-edit or reviewer corrections are tagged by error dimension + severity** (not just "right/wrong"), which lets an organization aggregate error-type frequency over time — e.g., seeing that "Terminology/Major" errors dominate points at a systemic terminology-consistency problem (fix the termbase), whereas a spike in "Fluency/Minor" suggests a style-guide gap, not a terminology gap. This diagnostic-by-category use is the whole rationale for a *typology* rather than a single pass/fail or numeric score. ([Multidimensional Quality Metrics (MQM) Framework — Emergent Mind](https://www.emergentmind.com/topics/multidimensional-quality-metrics-mqm), accessed 2026-07-13; [DQF-MQM error typology — TWB forum](https://community.translatorswb.org/t/dqf-mqm-error-typology/10827), accessed 2026-07-13)

## 6. AI/LLM translation steering practice

### 6.1 Glossary constraints in production MT/LLM pipelines

- **Google Cloud Translation Advanced** glossaries are consumed at translation time as either unidirectional term pairs or multilingual equivalent-term sets, imported from TSV/CSV/TMX. ([Creating and using glossaries — Google Cloud docs](https://docs.cloud.google.com/translate/docs/advanced/glossary), accessed 2026-07-13)
- **Amazon Translate** custom terminology is likewise supplied as CSV/TSV/TMX and is applied whenever the engine finds a source-text match against the terminology file, substituting the specified target term; a 2021 update added **multidirectional** terminology (one file serving many language-pair directions). ([Creating a custom terminology — Amazon Translate](https://docs.aws.amazon.com/translate/latest/dg/creating-custom-terminology.html), accessed 2026-07-13; [Amazon Translate now enables multidirectional custom terminology — AWS](https://aws.amazon.com/about-aws/whats-new/2021/11/amazon-translate-enables-multidirectional-custom-terminology), accessed 2026-07-13)
- **LLM-specific pattern — moving glossary enforcement out of the prompt:** current best practice for LLM-based translation treats the glossary as a **hard constraint layer separate from prompt text** rather than relying on the model to "remember" glossary terms embedded in a natural-language instruction — because prompt-only glossary mentions are not reliably honored at scale. Two concrete mechanisms are documented: (a) **constrained/guided decoding** — masking the next-token distribution at each generation step so the model literally cannot emit a token sequence that violates the schema/term constraint; (b) **retrieval-augmented term injection** — identifying key terms in the source segment and pulling their approved translations from a bilingual dictionary/TM into the prompt context via RAG, rather than shipping the whole glossary in every prompt. ([AI Translation with Glossary Support — Lokalise blog](https://lokalise.com/blog/ai-translation-glossary/), accessed 2026-07-13; [Refining Translations with LLMs: A Constraint-Aware Iterative Prompting Approach — arXiv](https://arxiv.org/html/2411.08348v1), accessed 2026-07-13; [Constrained decoding — zeroentropy.dev](https://zeroentropy.dev/concepts/constrained-decoding/), accessed 2026-07-13)

### 6.2 Few-shot example selection from TM for LLM prompting

Documented pattern: rather than hand-authoring few-shot examples, production LLM-translation prompt-engineering pulls **relevant TM matches and glossary hits and injects them dynamically into the prompt** as in-context examples — described as "dynamic context engineering… programmatically injecting relevant excerpts from Translation Memories (TMs) and glossary matches directly into the prompt structure" to reach enterprise-grade consistency beyond naive zero-shot prompting. ([Prompt Engineering for Translation: Guiding AI for Domain Accuracy — Translated](https://translated.com/resources/prompt-engineering-for-translation-guiding-ai-domain-accuracy), accessed 2026-07-13; corroborating claim also in [AI Translation with Glossary Support — Lokalise](https://lokalise.com/blog/ai-translation-glossary/), accessed 2026-07-13)

### 6.3 Adaptive MT that learns from post-edits — ModernMT

**Claim:** ModernMT (Translated/FBK/University of Edinburgh/TAUS, EU-funded) is the most concretely documented example of adaptive MT: it maintains a broad **Background Model** (trained on general-domain billions of sentence pairs across 60+ language pairs) plus a per-project **Foreground Model** built dynamically from the user's TM, glossaries, and live post-edit corrections. Each correction is incorporated **within milliseconds**, so subsequent segment suggestions immediately reflect the just-made correction — a "virtuous cycle" reducing post-editing effort as the session progresses. ([Adaptive Neural Machine Translation — Translated](https://translated.com/adaptive-neural-machine-translation), accessed 2026-07-13; [ModernMT — user guide — Matecat](https://guides.matecat.com/modernmt-mmt-plug-in), accessed 2026-07-13; [Adaptive Neural Machine Translation: How ModernMT Works — Translated](https://translated.com/resources/adaptive-neural-machine-translation-how-modernmt-works), accessed 2026-07-13; open-source repo: [modernmt/modernmt — GitHub](https://github.com/modernmt/modernmt), accessed 2026-07-13)

### 6.4 System-rules / instruction patterns

The recurring architectural pattern across the Lokalise, Translated, and constrained-decoding sources above is a **layered instruction stack**: (1) a stable "system rules" layer (tone/register, DNT list, forbidden-term list — rarely changed per-request), (2) a per-request **dynamic context layer** (TM fuzzy matches + glossary hits relevant to *this* segment, injected fresh each call), and (3) the actual source segment to translate. This mirrors, at the prompt-engineering level, the same separation CAT tools have long made between project-level settings (TM/termbase selection, QA profile) and segment-level state.

## 7. Bible-translation-specific practice

### 7.1 unfoldingWord translationWords (tW) as the terminology backbone

**Claim:** translationWords is explicitly a **key-term consistency tool**: it groups every occurrence of a recurring biblical concept so a translator/checker can compare the target-language rendering of that concept side-by-side across all its occurrences — functionally the same job a termbase-plus-consistency-QA-check does in secular CAT tooling, but curated manually.
**Evidence:** "The purpose of translationWords is to provide translators with concise definitions of important or difficult biblical concepts, along with translation suggestions... to help them make sound translation decisions," organized in three subfolders — `kt` (key terms — special theological importance), `other` (terms needing explanation but less central), and `names` (proper nouns). Term categories flagged as needing special translation attention: modern-but-different-ancient-usage terms (altar, priest, bless, clean/unclean, church); Bible-unique terms (Ark, gentile, tabernacle); transliterated terms (shekel, ephah, amen, apostle); and ambiguous terms where one Gateway Language word covers multiple underlying concepts (call, fear, age, great). ([unfoldingWord/translationWords — GitHub](https://github.com/unfoldingWord/translationWords), accessed 2026-07-13)

### 7.2 Paratext's Biblical Terms tool — the incumbent consistency-check practice

**Claim:** Paratext (the dominant desktop tool for Bible translation teams) has a dedicated **Biblical Terms** module that is the closest existing analogue to a terminology-consistency QA check, purpose-built for Scripture.
**Evidence:** The Biblical Terms tool and rendering window let a team "quickly check for accurate and consistent translation in all verses where a term occurs" and flag verses where the rendering deviates from the term's other occurrences. A documented **4-step process** for achieving consistency: review the verses where a term occurs, identify all renderings currently in use, decide the approved rendering(s), then clean up deviations. ([Translate Consistently with Key Biblical Terms — Paratext](https://paratext.org/features/biblical-terms/), accessed 2026-07-13; [10. A 4-Step process for consistent Biblical Terms — Paratext Manual](https://manual.paratext.org/10.BT/), accessed 2026-07-13)

Notably, Paratext explicitly supports **multiple valid renderings per term** (a single biblical term can legitimately have several different target-language renderings depending on sense/context) — closer to TBX's "more than one preferred term is allowed" model than to a strict one-term-one-translation rule. Translators can attach a parenthetical comment to a rendering to record *why* a given rendering was chosen for that sense, and the tool ignores parenthetical text when matching. ([Understanding the Challenge of Multiple Renderings for Biblical Terms — Paratext](https://paratext.org/2023/11/14/multiple-renderings-biblical-terms/), accessed 2026-07-13)

### 7.3 translationAcademy (tA) article naming/slug convention

**Claim:** tA articles are organized into named categories with a `category-shortname` slug pattern (e.g., `figs-metaphor` under the "Figures of Speech" category, alongside `grammar-`, `translate-`, and `writing-` prefixed slugs), referenced from other resources via `rc://` Resource Container links, e.g. `[[rc://en/ta/man/translate/figs-metaphor]]`. tA is distributed as a Markdown-based Resource Container (RC) with a manifest, per the RC Manual spec. ([unfoldingWord® Translation Academy — door43.org](https://door43.org/u/unfoldingWord/en_ta/master/03-translate.html), accessed 2026-07-13; corroborating repository: [unfoldingWord/en_ta README — Door43](https://git.door43.org/unfoldingWord/en_ta/src/branch/master/README.md), accessed 2026-07-13)

### 7.4 The Gateway Language Manual — style/register guidance for translating check-helps

This is the strongest, most directly relevant primary source found for how a Gateway Language (GL) team is instructed to translate/adapt the very resources (tN, tW, tQ) this panel will help manage.

**translationNotes (tN) guidance** (from the GL Manual's Gateway Language Translation page):
- Always translate the Notes with the GLT (Gateway Language Text) and GST (Gateway Simplified Text) — and the source English Notes — visible simultaneously.
- The bolded/quoted snippet in a note must be **copied directly** from the Gateway Language text (not re-translated from English).
- Alternate translations offered in a note must be grammatically substitutable directly into the GL sentence.
- Notes must be **adapted**, not translated literally, when the target GLT phrases something differently than the English ULT: "you will need to adapt the note so that it talks about the text as it is in the GLT."
- Notes that don't apply to the target GLT's phrasing should be **deleted**; new notes should be **added** when the target GL's own grammar/idiom needs explanation that English didn't need.
- Recommended translator qualification: training in linguistics as well as biblical studies.
([Gateway Language Translation — Gateway Language Manual](https://gl-manual.readthedocs.io/en/latest/gl_translation.html), accessed 2026-07-13)

**translationWords (tW) guidance** (same source): use Bible-context definitions, not modern-dictionary definitions, since biblical usage of a word often differs from everyday usage; list multiple GL-language words per entry when the target language splits a concept the source language doesn't; drop a suggested rendering if it doesn't work in the GL rather than forcing it; translate section headers once, not repeatedly per occurrence.

**translationQuestions (tQ) guidance** (same source): translate only the question; the **answer must be copied verbatim from the already-translated GLT** rather than independently translated, specifically to guarantee the answer stays consistent with the actual Scripture rendering.

**Book Package Style Sheet** (Appendix A of the same manual) covers: mandatory camelCase "unfoldingWord" branding; en-dash (not hyphen) for verse ranges (e.g. "Genesis 1:1–5"); no comma between a citation and its version abbreviation ("Job 4:8 ULT"); American-English quotation-mark nesting and comma-placement rules; Oxford/series comma; spelled-out numbers one–ten vs. numerals above ten; single (not double) spaces between sentences; no contractions; formal register (who/whom distinction, avoid split infinitives); capitalize "Scripture(s)" and "Word" when referring to the Bible/Jesus. ([Appendix A. unfoldingWord Book Package Style Sheet — Gateway Language Manual](https://gl-manual.readthedocs.io/en/latest/gl_appendixA.html), accessed 2026-07-13)

**English content-editing guidelines** (`gl_guidelines.html` in the same manual) additionally cover: audience is defined as **English-second-language speakers with roughly a high-school education level** — so avoid specialist linguistic jargon in notes (e.g. say "first-level quotation" rather than a more technical linguistics term), avoid idioms unfamiliar to non-native speakers, and ensure every note is **actionable** (a translator should know concretely what to do differently after reading it). Notes should bold the source-text snippet under discussion, use the fixed phrase "Alternate translation:" for optional renderings, and are capped at **one tA article reference per note**. ULT and UST are deliberately kept lexically distinct from each other (e.g., ULT "covenant faithfulness" vs. UST "kind/faithful"; ULT "gospel" vs. UST "good news") so that the two source texts give translators two independent windows onto the meaning rather than one paraphrased twice. *(Note: one line attributed in the fetched summary to this page — a "Humble, not heroic" brand-voice phrasing — duplicates unfoldingWord's general brand-voice materials closely enough that it could not be independently re-verified as page content during this research and is deliberately excluded from this citation to avoid overclaiming; the rest of the guidance above was corroborated by the structure and specificity of the fetched content and is reported with **medium-high confidence**.)* ([Guidelines for English Language Content Editing — Gateway Language Manual](https://gl-manual.readthedocs.io/en/latest/gl_guidelines.html), accessed 2026-07-13)

### 7.5 FOBAI (Forum of Bible Agencies International)

**Claim:** FOBAI is an alliance of 25+ Bible agencies that has adopted cross-agency standards, but its published standards are at the level of **translation principles/procedures and translator/consultant qualifications**, not terminology-tooling or TM/QA-check specifics.
**Evidence:** FOBAI's "Basic Principles and Procedures for Bible Translation" (approved April 2017) and its "Qualifications and Standards" pages define what counts as an acceptable translation process and who is qualified to consult/check it; the organization notes it is *considering* developing standards in adjacent areas (church-collaboration best practice, audio/video, sign-language, orality) but nothing was found describing a shared terminology database, TM interchange practice, or QA-check taxonomy at the FOBAI level. ([Translation Standards — Forum of Bible Agencies International](https://forum-intl.org/resources/translation-standards/), accessed 2026-07-13; [Basic Principles and Procedures for Bible Translation, FOBAI, April 2017 (PDF)](https://forum-intl.org/wp-content/uploads/2019/03/FOBAITranslationBasicPrinciplesandProceduresApril2017.pdf), accessed 2026-07-13)

- **A GL-manual passage on formality/register control specifically for AI-assisted drafting** (as opposed to human-translator register guidance) was not found; the GL Manual's register guidance (§7.4) predates AI-assisted drafting and is written for human translators only.

## 9. Design implications

Distilled, actionable takeaways for the Phase 2 panel design, each traceable to a section above:

1. **Term status should be a small, closed picklist**, not free text — mirror the TBX-standard core: Preferred / Admitted / Deprecated / Superseded, plus a tool-layer (non-TBX) **Forbidden** and a separate **Do-Not-Translate** flag, since DNT is semantically distinct from "forbidden" (§2.2–2.3).
2. **Forbidden-term entries need a paired "use instead" field.** A bare forbidden flag without a replacement suggestion forces the translator to guess; every CAT tool with forbidden-term support pairs it with visible guidance (§2.3 — flagged as inference, but low-risk to build).
3. **Termbase entries should be concept-oriented, not just word-pairs** — one concept can have multiple valid target-language renderings attached (this matches both TBX's structure and Paratext's explicit "multiple renderings per term" model, which is directly relevant given translationWords already works this way) (§2.1, §7.2).
4. **TM/reuse matching (if built at all) needs at least three tiers**: exact/100%, context-exact (ICE/101%+, no review needed), and fuzzy-below-100% with a configurable minimum threshold (commonly ~70–75%) and a penalty mechanism for matches sourced from lower-trust origin (e.g., machine-drafted vs. human-approved) (§1.2).
5. **QA severities converge on 2–3 tiers**: a non-blocking "warning" (visible, overridable/dismissible with a reason) and a blocking "error" (cannot save/submit); memoQ's richer model adds penalty points per severity for aggregate QA scoring. Building exactly 2–3 severities (not 5+) matches the dominant convention (§4.2).
6. **Every non-blocking QA flag should support an explicit "ignore with reason" action** — this is universal across Smartling/Crowdin/memoQ and is what keeps QA from becoming an unusable gate (§4.2).
7. **QA checks should cover the same core categories the industry has converged on**: terminology/forbidden-term enforcement, source→target consistency (and the reverse — same target used for different sources), numbers/tags/placeholders, punctuation, segment length, and untranslated content (§4.1) — directly portable to a tN/tW/tQ-editing context (e.g., "same tA reference cited with contradictory guidance," "same key term rendered two different ways across notes").
8. **A style/register control should be a small closed enum**, following DeepL's and Amazon's precedent (Default/Formal/Informal, or similar), rather than free-text tone description — this is the one formality convention already standardized at the API level across two major MT vendors (§3.3).
9. **A "translation brief" concept (purpose, audience, register, terminology pointer) is worth building as a first-class object**, not just implicit settings — ISO 17100 and LSP practice treat it as a discrete artifact that travels with a project, and it directly maps onto what a GL team already needs to declare for a book/project (§3.1).
10. **Style-guide content should separate "voice/tone statement" from "controlled word list"** from "localization-specific rules" — this is the structure both Microsoft's and Google's public guides use, and it maps cleanly onto unfoldingWord's own Book Package Style Sheet (branding, punctuation, register) (§3.2, §7.4).
11. **LQA/error-classification, if built, should use a small fixed dimension set** (Accuracy / Fluency / Terminology / Style, at minimum) **with an orthogonal severity axis**, not a single flat "quality score" — this is what makes error data usable for process improvement (spotting whether a recurring problem is terminological vs. stylistic) (§5).
12. **If AI-drafted content is scored/reviewed, keep the same dimension+severity model uniform across human QA and AI-draft QA** — the payoff of MQM/DQF-style typologies is comparability across correction sources over time; a bespoke AI-only rubric would forfeit that (§5.3, §6).
13. **Glossary/terminology enforcement for the AI pipeline should be a constraint layer, not prompt text** — if/when this repo's AI pipeline (uw-bt-bot) is steered by a termbase, the durable industry pattern is enforcing terms via retrieval/constraint mechanisms rather than trusting a model to honor a glossary pasted into a system prompt (§6.1). This is an important flag for whoever designs the AI-facing side of this panel, since the pipeline itself is external to this repo (§8, gap noted).
14. **Few-shot/context injection for AI drafting should pull dynamically from TM + glossary matches relevant to the current segment**, not a static example set — mirrors both ModernMT's per-segment adaptation and the "dynamic context engineering" pattern documented for LLM translation prompting (§6.2–6.3).
15. **CSV/TSV import for glossaries should follow the cross-vendor convention**: UTF-8, header row naming columns, source term immediately adjacent to target term, one consistent language-code format per file — this is portable almost verbatim from Google/Amazon/DeepL glossary-import conventions (§2.4).
16. **Respect the existing GL Manual conventions for tN/tW/tQ translation** rather than inventing new register rules — e.g., tQ answers must be copied verbatim from the GLT (not independently translated), tN alternate-translations must be grammatically substitutable, and tW definitions must be Bible-context-specific. Any AI-assisted drafting or QA feature for these resource types should be checked against these existing rules rather than importing secular localization norms wholesale (§7.4).
17. **Key-term consistency should stay concept-oriented and allow multiple valid renderings**, following both TBX and Paratext's explicit multi-rendering model — a strict one-term-one-target-string rule would fight against how translationWords and Paratext's Biblical Terms tool already work, and against how Bible translators actually work with sense-dependent renderings (§2.1, §7.1–7.2).
18. **A rendering-comment field (parenthetical, ignored by matching) is worth replicating** from Paratext's Biblical Terms tool — it's a cheap, already-validated UX pattern for recording *why* a particular rendering was chosen for a particular sense, without breaking automated consistency matching (§7.2).
19. **TM maintenance, if TM is built at all, needs an explicit "penalize this source" and "prune stale entries" workflow** from day one — these are described industry-wide as routine (often quarterly/annual) maintenance, not optional nice-to-haves, because uncurated TM/AI-draft history actively degrades match quality over time (§1.3).
20. **A CAT-tool QA profile and an LQA typology don't have to be separate systems** — Phrase/Memsource's LQA module is built directly on the DQF-MQM typology and lets weights vary per content type (§4.1, §5.2). If this panel eventually wants both "everyday QA checks" and "formal error scoring," reusing one dimension+severity model for both (rather than inventing a second taxonomy) avoids exactly the double-bookkeeping Phrase's design already solved.
21. **Treat the AI pipeline's actual prompt/glossary-steering mechanism as an open question to confirm, not an assumption** — this research found strong general-industry practice (§6) but no documentation of what unfoldingWord's own `uw-bt-bot` pipeline currently does; Phase 2 design should include a step to check with the pipeline's maintainers before assuming glossary/TM data entered in this panel will actually reach the AI pipeline in a usable form (§8).
