2026-07-20 · **worktree-dcs-teams-roles** — **Door43 teams are now a role source (read-side); the last deferred auth item from the demo-prep list.**
- In plain terms: instead of every editor having to be added by hand in the Preferences panel, you can now put people in a Door43 team and they get the matching role in BibleEditor the next time they sign in. Adding someone by hand still works and still wins — teams only ever *add* access on top of what an admin granted directly.
- At OAuth callback (`api/src/auth.ts`) we call `GET /api/v1/user/teams` with the **user's own** access token (no service-token scope needed), filter to teams inside `getProjectConfig().org`, and map name → role: `BE-Admins` → admin, `BE-Editors` → editor. Both names are env-overridable (`DCS_TEAM_ADMIN` / `DCS_TEAM_EDITOR`) because the *actual* team names in the `BibleEditorMLTest` test org could not be read from this session — the Gitea teams API requires auth and no Chrome browser was connected.
- New `api/src/dcsTeams.ts` (paginated fetch, pure name→role mapping, D1 sync) + migration `0053_user_roles_source.sql` adding `user_roles.source` (`'manual'` | `'dcs_team'`). Caching into `user_roles` is what keeps `/api/auth/refresh` a plain D1 read — no DCS round-trip per refresh. `viewer` stays dynamic (org membership), never cached.
- Precedence is one rule: **a row belongs to whoever created it, and teams may only ever raise access on rows they don't own.** Team rows track their team in both directions; manual rows can be promoted by a team (editor → admin, which is how legacy pre-0053 allowlist entries get handed to team control) but never demoted or deleted; `source` never changes after insert. The last remaining admin is never demoted *or* deleted. Anything leaving membership unknown — network error, non-2xx, unparseable body, or a page-cap-truncated list — skips the sync instead of reading as "on no teams", and logs.
- **A high-effort review pass caught 10 confirmed issues in the first cut of this, all fixed here.** The load-bearing ones: the demote path had no last-admin guard (moving the sole admin from BE-Admins to BE-Editors emptied the admin set, and `/api/admin/users` is admin-gated, so recovery needed raw SQL); the callback block was unguarded, so deploying before applying migration 0053 would 500 *every* sign-in; and the org came from `getProjectConfig`, which silently falls back to unfoldingWord on a read error — a transient D1 hiccup would have revoked every GL project's team roles and could have granted admin to an unfoldingWord BE-Admins member. Also fixed: a page-cap-truncated team list was treated as authoritative; admin-panel edit/remove of a team row silently didn't stick; and team-role revocation lagged up to 14 days because refresh never re-checked.
